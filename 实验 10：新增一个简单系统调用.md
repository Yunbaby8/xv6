# 实验 10：新增一个简单系统调用

## 一、实验目的

本实验的目标是在 Linux 内核中新增一个简单的系统调用，并在用户态程序中调用它。

本次实现的系统调用功能很简单：

```text
用户态程序调用 syscall(548)
内核执行我们自己写的 my_syscall
返回固定整数 42
```

通过这个实验，可以理解：

```text
用户态程序如何进入内核态
系统调用号如何对应到内核函数
syscall table 的作用
内核代码如何参与编译
用户态如何测试自己新增的 syscall
```

这个实验的重点不是返回 `42` 本身，而是理解：

```text
用户态 → syscall 入口 → syscall table → 内核函数 → 返回用户态
```

## 2. 新增系统调用实现文件

新建文件：

```bash
nano kernel/my_syscall.c
```

写入代码：

```c
#include <linux/kernel.h>
#include <linux/syscalls.h>

SYSCALL_DEFINE0(my_syscall)
{
    printk(KERN_INFO "[my_syscall] called\n");
    return 42;
}
```

保存退出。

## 3. 修改 kernel/Makefile

打开：

```bash
nano kernel/Makefile
```

在文件末尾加入：

```make
obj-y += my_syscall.o
```

这一步的作用是告诉内核编译系统：

```text
请把 kernel/my_syscall.c 编译进内核
```

否则你虽然写了 `.c` 文件，但它不会真正进入内核镜像。

## 4. 修改 syscall table

打开 x86_64 的系统调用表：

```bash
nano arch/x86/entry/syscalls/syscall_64.tbl
```

在文件末尾加入一行：

```text
548    common    my_syscall    sys_my_syscall
```

这行的含义是：

```text
548              系统调用号
common           ABI 类型
my_syscall       系统调用名字
sys_my_syscall   内核中真正执行的函数
```

用户态调用：

```c
syscall(548);
```

内核就会根据这个表找到：

```text
sys_my_syscall
```

然后执行它。

# 五、代码解析

## 1. `#include <linux/kernel.h>`

```c
#include <linux/kernel.h>
```

这是内核里的常用头文件。

这里主要是为了使用一些内核基础功能，比如：

```c
printk()
```

注意，内核里不能用普通用户态的：

```c
printf()
```

内核里打印日志要用：

```c
printk()
```

------

## 2. `#include <linux/syscalls.h>`

```c
#include <linux/syscalls.h>
```

这个头文件提供了定义系统调用需要的宏，比如：

```c
SYSCALL_DEFINE0
SYSCALL_DEFINE1
SYSCALL_DEFINE2
```

本实验使用：

```c
SYSCALL_DEFINE0(my_syscall)
```

------

## 3. `SYSCALL_DEFINE0(my_syscall)`

```c
SYSCALL_DEFINE0(my_syscall)
{
    return 42;
}
```

这里的 `0` 表示这个系统调用没有参数。

可以理解成：

```text
定义一个没有参数的系统调用，名字叫 my_syscall
```

它不是普通函数写法，但它最终会帮你生成内核真正调用的函数。

简单理解：

```c
SYSCALL_DEFINE0(my_syscall)
{
    return 42;
}
```

大概等价于：

```c
long sys_my_syscall(void)
{
    return 42;
}
```

所以 syscall 表里最后一列要写：

```text
sys_my_syscall
```

------

## 4. `printk(KERN_INFO "...")`

```c
printk(KERN_INFO "[my_syscall] called\n");
```

这是内核打印日志。

它不会直接打印到普通终端，而是进入内核日志缓冲区。

所以要用：

```bash
dmesg | tail
```

查看。

------

## 5. `return 42;`

```c
return 42;
```

这是内核系统调用返回给用户态程序的值。

用户态程序这里接收：

```c
long ret = syscall(__NR_my_syscall);
```

所以最终：

```c
printf("my_syscall return = %ld\n", ret);
```

会输出：

```text
my_syscall return = 42
```

------

## 6. syscall 表这一行

```text
548    common    my_syscall    sys_my_syscall
```

这一行非常关键。

它把系统调用号和内核函数连起来。

可以理解成一个映射表：

```text
548  →  sys_my_syscall
```

当用户态执行：

```c
syscall(548);
```

内核就会找到第 548 号对应的内核处理函数：

```text
sys_my_syscall
```

然后执行它。

------

## 7. 用户态测试代码解析

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/syscall.h>
```

这些是用户态头文件。

其中：

```c
#include <sys/syscall.h>
```

提供了 `syscall()` 的相关声明。

------

```c
#define __NR_my_syscall 548
```

这里定义系统调用号。

这个数字必须和内核 syscall 表中的数字一致：

```text
548    common    my_syscall    sys_my_syscall
```

如果这里写错，用户态就会调用错系统调用，或者直接失败。

------

```c
long ret = syscall(__NR_my_syscall);
```

这句话是真正发起系统调用。

用户程序不是直接调用：

```c
my_syscall();
```

因为 `my_syscall` 在内核里，用户程序看不到。

用户程序只能通过系统调用号进入内核。

# 六、完整调用链

这个实验最重要的是理解这条链：

```text
用户态程序
    |
    | syscall(548)
    v
CPU 从用户态进入内核态
    |
    v
内核 syscall 入口
    |
    v
查 syscall_64.tbl
    |
    v
548 对应 sys_my_syscall
    |
    v
执行 SYSCALL_DEFINE0(my_syscall) 定义的代码
    |
    v
return 42
    |
    v
返回用户态
    |
    v
printf 打印 42
```

简化成一句话：

```text
syscall(548) → syscall_64.tbl → sys_my_syscall → return 42
```

# 七、容易混乱的地方

## 1. `my_syscall.c` 是文件名，不是 syscall 名的本质

```text
kernel/my_syscall.c
```

只是放代码的文件。

真正重要的是里面的：

```c
SYSCALL_DEFINE0(my_syscall)
```

文件名可以叫别的，但为了清楚，建议和 syscall 名保持一致。

------

## 2. `my_syscall` 和 `sys_my_syscall` 不是同一个东西

```c
SYSCALL_DEFINE0(my_syscall)
```

里面的：

```text
my_syscall
```

是你给系统调用起的名字。

内核宏会根据它生成：

```text
sys_my_syscall
```

所以关系是：

```text
my_syscall → sys_my_syscall
```

也就是自动加了 `sys_` 前缀。

------

## 3. `548` 是系统调用号，不是函数名

用户态程序调用的是：

```c
syscall(548);
```

不是：

```c
my_syscall();
```

因为用户态不能直接调用内核函数。

系统调用号就像编号：

```text
548 号 syscall 对应 sys_my_syscall
```

------

## 4. 为什么要改 `kernel/Makefile`

因为内核不是看到 `.c` 文件就自动编译。

你新建了：

```text
kernel/my_syscall.c
```

还必须告诉编译系统：

```make
obj-y += my_syscall.o
```

否则它不会进入最终的内核镜像。

------

## 5. 为什么测试程序要放进 initramfs

QEMU 里的最小 Linux 和你的宿主机不是同一个文件系统。

你在宿主机有：

```text
/home/yunhe/allcode/kernel-lab/initramfs/test_my_syscall
```

打包后，QEMU 里才会出现：

```text
/test_my_syscall
```

所以用户态测试程序要放进：

```text
kernel-lab/initramfs
```

然后重新打包。

------

## 6. 为什么要先编译测试程序再打包

`test_my_syscall.c` 只是源代码，不能直接运行。

必须先编译成可执行文件：

```bash
gcc test_my_syscall.c -static -o test_my_syscall
```

然后再打包 initramfs。

正确顺序是：

```text
写 test_my_syscall.c
    ↓
gcc 编译成 test_my_syscall
    ↓
pack.sh 打包进 initramfs.cpio
    ↓
QEMU 启动
    ↓
运行 ./test_my_syscall
```

------

## 7. 放进 initramfs 后不会自动执行

程序放进 initramfs 只是说明 QEMU 里的 Linux 能找到它。

默认自动执行的是：

```text
/init
```

不是：

```text
/test_my_syscall
```

所以启动后还要手动运行：

```bash
./test_my_syscall
```

如果想自动运行，需要修改 initramfs 里的 `/init` 脚本。

------

## 8. `printk` 不是 `printf`

用户态用：

```c
printf()
```

内核态用：

```c
printk()
```

所以内核代码里不能写：

```c
printf("hello\n");
```

要写：

```c
printk(KERN_INFO "hello\n");
```

查看日志用：

```bash
dmesg
```

------

# 八、常见错误和排查方法

## 1. 运行结果不是 42

检查用户态 syscall 号是否和内核表一致。

内核表：

```text
548    common    my_syscall    sys_my_syscall
```

用户程序：

```c
#define __NR_my_syscall 548
```

两边必须一样。

------

## 2. 编译内核报错

重点检查：

```text
kernel/my_syscall.c
kernel/Makefile
arch/x86/entry/syscalls/syscall_64.tbl
```

尤其是 syscall 表最后一列和 `SYSCALL_DEFINE0(my_syscall)` 的名字是否对应。

------

## 3. QEMU 里找不到 test_my_syscall

说明你可能没有重新打包 initramfs。

重新执行：

```bash
cd /home/yunhe/allcode/kernel-lab
./scripts/pack.sh
```

然后重新启动 QEMU。

------

## 4. dmesg 里看不到 `[my_syscall] called`

可能原因：

```text
没有运行 ./test_my_syscall
没有启动新编译的内核
syscall 号不对
syscall 表没改对
```

可以检查 QEMU 是否使用新的 bzImage。

# 九、这个实验对学习 Linux 内核有什么帮助

这个实验虽然代码很少，但意义很大。

## 1. 理解用户态和内核态的边界

普通程序运行在用户态。

内核运行在内核态。

用户程序不能随便访问硬件、内核数据结构、内核函数。

所以必须通过系统调用进入内核。

这个实验让你真正看到：

```text
用户态程序不能直接调用内核函数
必须通过 syscall 号进入内核
```

------

## 2. 理解 syscall 是 Linux 的核心接口

Linux 里很多常见操作本质上都依赖系统调用，比如：

```text
open
read
write
fork
execve
mmap
ioctl
```

你自己新增 syscall 后，就更容易理解这些函数背后的原理。

例如：

```c
printf("hello\n");
```

表面上是打印，底层最终会走到类似：

```text
write syscall
```

------

## 3. 理解 syscall table 的作用

以前可能只知道系统调用是“用户程序请求内核”。

做完这个实验后，你会知道它实际需要一个表来映射：

```text
系统调用号 → 内核函数
```

这就是 syscall table 的作用。

------

## 4. 理解内核编译系统

这个实验还训练了你对内核 Makefile 的理解。

你知道了：

```text
写了 .c 文件，不代表它会被编译
必须加到 Makefile / Kbuild 系统里
```

这对以后学习驱动、文件系统、网络模块都有帮助。

------

## 5. 理解用户态测试和内核功能验证

内核代码不能像普通 C 程序那样直接运行。

你需要：

```text
修改内核
编译内核
启动新内核
写用户态程序
触发内核代码
查看返回值和 dmesg
```

这是非常典型的内核开发验证流程。

------

## 6. 为后续学习驱动打基础

以后学习字符设备驱动时，你会遇到：

```text
open
read
write
ioctl
mmap
```

这些本质上也都是用户态和内核态交互。

系统调用实验能帮你提前理解：

```text
用户态怎么请求内核
内核怎么返回结果
用户态和内核态为什么不能随便互相访问
```

------

## 7. 为理解 copy_from_user / copy_to_user 打基础

本实验只是返回一个整数，没有用户指针参数。

下一步如果做带参数 syscall，比如：

```c
sys_add(int a, int b)
```

或者：

```c
sys_get_msg(char __user *buf)
```

就会涉及：

```text
copy_from_user
copy_to_user
```

那时你会进一步理解：

```text
为什么用户态内存和内核态内存不能直接乱访问
```

------

# 十、一句话总结

这个实验完整打通了：

```text
用户态程序 → syscall 号 → syscall table → 内核函数 → 返回用户态
```

你通过新增一个 `my_syscall`，理解了 Linux 内核对外提供服务的基本方式。

它不是为了实现一个有用的功能，而是为了让你真正理解：

```text
用户程序到底是怎么进入 Linux 内核的
```

这是学习 Linux 内核、驱动开发、系统编程非常重要的一步。
