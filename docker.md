# 一、你从一开始做了哪些实验

## 1. 验证 Docker 是否能真正工作

你做了：

```bash
docker pull hello-world
docker run hello-world
```

后来因为权限问题，你用过 `sudo docker ...`。

### 这个实验在干什么

这是 Docker 最小闭环实验：

- 从仓库拉镜像
- 基于镜像创建容器
- 启动容器执行默认程序
- 把结果输出到终端

### 你看到的现象

- `hello-world` 镜像被拉到本地
- 容器运行后打印欢迎信息
- 任务执行完就退出

------

## 2. 查看本地镜像和容器记录

你做了：

```bash
docker images
docker ps -a
```

### 这个实验在干什么

分别看两类对象：

- **images**：本地有哪些镜像
- **containers**：有哪些容器被创建过，以及它们现在什么状态

### 你看到的现象

- 本地出现了 `hello-world` 镜像
- `docker ps -a` 里能看到运行过的 `hello-world` 容器，状态是 `Exited (0)`

------

## 3. 进入 Ubuntu 容器，创建文件，再验证容器停掉后文件还在

你做了大概这样的实验：

```bash
docker run -it --name mytest ubuntu bash
echo "hello from container" > /tmp/test.txt
cat /tmp/test.txt
exit
docker ps -a
docker start mytest
docker exec -it mytest bash
cat /tmp/test.txt
```

### 这个实验在干什么

验证容器文件系统和宿主机文件系统的关系：

- 文件默认写在容器自己的可写层里
- 容器停止后，文件通常还在
- 宿主机普通目录默认看不到容器里的这个文件

### 你看到的现象

- 在容器里创建的 `/tmp/test.txt` 能重新进入同一个容器后再次看到
- 在宿主机直接 `cat /tmp/test.txt` 看不到这个文件

------

## 4. 运行 nginx 容器，做端口映射，并从宿主机访问它

你做了类似：

```bash
docker pull nginx
docker run -d --name mynginx -p 8080:80 nginx
docker ps
curl http://127.0.0.1:8080
docker logs mynginx
docker stop mynginx
docker ps -a
```

### 这个实验在干什么

验证服务型容器和端口映射：

- 容器里可以跑持续运行的服务
- 宿主机可以通过端口映射访问容器里的服务
- 容器日志可以查看
- 容器可以停止

### 你看到的现象

- `mynginx` 是 `Up` 状态
- `curl 127.0.0.1:8080` 返回了 nginx 默认网页
- `logs` 里能看到请求记录
- `stop` 后容器状态变成 `Exited`

------

## 5. 进入运行中的 nginx 容器内部查看文件

你做了：

```bash
docker exec -it mynginx sh
pwd
ls
ls /usr/share/nginx/html
cat /usr/share/nginx/html/index.html
```

### 这个实验在干什么

验证“宿主机看到的网页”和“容器内部文件”之间的关系。

### 你看到的现象

- 容器内部有自己的 Linux 目录结构
- nginx 默认网页目录在 `/usr/share/nginx/html`
- `index.html` 的内容和你从宿主机 `curl` 到的网页内容对应

------

## 6. 用挂载目录替换 nginx 默认页面

你做了挂载实验，大概类似：

```bash
mkdir -p ~/myweb
# 写自己的 index.html
docker stop mynginx
docker rm mynginx
docker run -d --name mynginx -p 8080:80 -v ~/myweb:/usr/share/nginx/html nginx
curl http://127.0.0.1:8080
```

### 这个实验在干什么

验证宿主机目录挂载进容器后，容器服务可以直接使用宿主机文件。

### 你看到的现象

- nginx 默认页面被你宿主机目录里的页面替换了
- 说明容器里对应目录被挂载目录“盖住”了

------

# 二、你学会了哪些 Docker 核心概念

## 1. 镜像 image 和容器 container 不是一回事

### 你学会了什么

- **镜像**是模板、环境基础
- **容器**是镜像运行后的实例

### 在 Docker 中有什么作用

这是 Docker 最核心的对象模型。
后面你做任何事都绕不开这两个对象：

- 拉镜像
- 基于镜像起容器
- 看容器状态
- 删容器
- 构建新镜像

如果这两个概念不清，后面 Dockerfile、挂载、构建环境都会乱。

------

## 2. `docker run` 本质上是“创建并启动容器”

### 你学会了什么

不是简单“运行一句命令”，而是：

**run = create + start**

### 在 Docker 中有什么作用

你以后看到任何 `docker run ...`，都知道它背后是在：

- 基于某个镜像创建一个新容器
- 然后启动它

这能帮你理解为什么运行后 `ps -a` 里会多一个新容器。

------

## 3. 容器有自己的文件系统视图

### 你学会了什么

- 容器里有自己的目录结构
- 容器里写文件，默认写在容器自己的可写层
- 宿主机默认看不到这些文件

### 在 Docker 中有什么作用

这直接关系到以后你理解：

- 数据会不会丢
- 代码应该放哪
- 为什么要挂载目录
- 为什么删容器后有些数据没了

------

## 4. 停容器不等于删容器

### 你学会了什么

- 容器停了，容器实例还在
- 同一个容器重新启动后，之前的文件通常还在
- 真正删掉容器，那层可写数据通常才没了

### 在 Docker 中有什么作用

以后你做实验、调试、开发时，会经常区分：

- `stop`
- `start`
- `rm`

这是操作容器生命周期的基础。

------

## 5. 容器里可以跑两类程序：一次性任务和持续服务

### 你学会了什么

- `hello-world` 属于一次性任务容器，跑完就退出
- `nginx` 属于服务型容器，会一直运行

### 在 Docker 中有什么作用

这帮你理解 Docker 不只是“打印一句话的工具”，它更常用来跑：

- Web 服务
- 数据库
- 后端服务
- 构建环境
- 开发工具

------

## 6. 端口映射 `-p` 的意义

### 你学会了什么

例如：

```bash
-p 8080:80
```

表示：

- 宿主机端口：8080
- 容器端口：80

宿主机访问 8080，相当于访问容器里的 80。

### 在 Docker 中有什么作用

这是让宿主机访问容器内服务的关键机制。
以后你跑任何服务型容器都可能用到：

- nginx
- 后端 API
- 数据库管理页面
- 前端项目
- 调试服务

------

## 7. 日志 `docker logs` 是排查容器的重要手段

### 你学会了什么

`docker logs mynginx` 能看到容器启动信息、请求记录、错误输出。

### 在 Docker 中有什么作用

以后你排查容器问题，最常见的就是看日志。
这比“删了重来”更像真正的开发/运维思路。

------

## 8. `docker exec` 和 `docker run` 的区别

### 你学会了什么

- `run`：新建并启动容器
- `exec`：进入已运行容器执行命令

### 在 Docker 中有什么作用

这是以后调试和检查运行中容器的基本能力。
比如：

- 进去看配置文件
- 进去看目录
- 进去手动执行命令
- 进去排查服务问题

------

## 9. 宿主机目录挂载 `-v` 的意义

### 你学会了什么

比如：

```bash
-v ~/myweb:/usr/share/nginx/html
```

表示把宿主机目录挂到容器目录。
从容器运行时视角看，就像把原目录内容“盖住”了。

### 在 Docker 中有什么作用

这是 Docker 开发环境最重要的能力之一。
因为真实开发里很常见这种模式：

- 代码在宿主机
- 环境在容器
- 通过挂载把代码给容器里的服务/工具用

这和你以后可能接触的 dockerized build 非常相关。

------

## 2. 你已经从“使用现成镜像”入门

- 拉镜像

- 看容器

- 做挂载

## 3. 这些能力和你之后可能接触的 OpenBMC dockerized build 是相关的

所以你现在学的不是无关的“通用命令”，而是在补以后很可能会用到的思维模式。
