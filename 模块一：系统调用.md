# Linux 自定义系统调用实验笔记

## 一、实验目的

本实验的目标是在 Linux 6.6.87 内核中添加自定义系统调用，并通过用户态程序进行测试，从而理解用户态程序如何通过 `syscall` 指令进入内核态，以及系统调用在内核中的注册、分发、执行和返回过程。

通过本实验，需要掌握以下内容：

```text
1. 系统调用号 syscall number 的作用
2. syscall_64.tbl 如何建立系统调用号和内核函数之间的映射
3. SYSCALL_DEFINE 宏如何定义系统调用
4. 用户态如何通过 syscall() 调用自定义系统调用
5. current 在系统调用中表示什么
6. printk 输出到哪里
7. 用户态和内核态之间为什么要用 copy_to_user / copy_from_user
8. 错误码如何从内核返回到用户态 errno
9. initramfs 环境下为什么用户程序需要静态编译
```

本实验实现了 4 个自定义系统调用：

| 系统调用号 | 系统调用名          | 功能                                          |
| ---------- | ------------------- | --------------------------------------------- |
| `549`      | `my_add`            | 测试普通参数传递和返回值                      |
| `550`      | `my_task_info`      | 测试 `current` 当前进程上下文                 |
| `551`      | `my_get_comm`       | 测试内核向用户态传数据，使用 `copy_to_user`   |
| `552`      | `my_print_user_msg` | 测试用户态向内核传数据，使用 `copy_from_user` |

## 2. 修改系统调用表

编辑文件：

```bash
vim arch/x86/entry/syscalls/syscall_64.tbl
```

在合适位置添加：

```text
549	common	my_add		sys_my_add
550	common	my_task_info	sys_my_task_info
551	common	my_get_comm	sys_my_get_comm
552	common	my_print_user_msg	sys_my_print_user_msg
```

这一张表的作用是告诉 x86_64 内核：

```text
系统调用号 549 对应 my_add
系统调用号 550 对应 my_task_info
系统调用号 551 对应 my_get_comm
系统调用号 552 对应 my_print_user_msg
```

以第一行为例：

```text
549    common    my_add    sys_my_add
```

含义是：

| 列           | 含义                                   |
| ------------ | -------------------------------------- |
| `549`        | 系统调用号                             |
| `common`     | ABI 类型，普通 x86_64 syscall 路径可用 |
| `my_add`     | 系统调用逻辑名字                       |
| `sys_my_add` | 系统调用入口符号名                     |

用户态调用：

```c
syscall(549, 10, 20);
```

内核就会根据 `549` 找到对应系统调用，并最终执行 `my_add` 的内核实现。

## 3. 添加内核系统调用实现文件

新建文件：

```bash
vim kernel/my_syscalls.c
```

完整代码如下：

```c
// kernel/my_syscalls.c

#include <linux/kernel.h>
#include <linux/syscalls.h>
#include <linux/sched.h>
#include <linux/uaccess.h>
#include <linux/errno.h>
#include <linux/string.h>

SYSCALL_DEFINE2(my_add, int, a, int, b)
{
	int result = a + b;

	printk(KERN_INFO "[my_add] current=%s pid=%d a=%d b=%d result=%d\n",
	       current->comm, current->pid, a, b, result);

	return result;
}

SYSCALL_DEFINE0(my_task_info)
{
	printk(KERN_INFO "[my_task_info] comm=%s pid=%d tgid=%d\n",
	       current->comm, current->pid, current->tgid);

	return current->pid;
}

SYSCALL_DEFINE2(my_get_comm, char __user *, buf, size_t, len)
{
	char comm[TASK_COMM_LEN];

	if (!buf) {
		printk(KERN_INFO "[my_get_comm] error: buf is NULL\n");
		return -EINVAL;
	}

	if (len < TASK_COMM_LEN) {
		printk(KERN_INFO "[my_get_comm] error: len=%zu is too small, need %d\n",
		       len, TASK_COMM_LEN);
		return -EINVAL;
	}

	get_task_comm(comm, current);

	if (copy_to_user(buf, comm, TASK_COMM_LEN)) {
		printk(KERN_INFO "[my_get_comm] error: copy_to_user failed\n");
		return -EFAULT;
	}

	printk(KERN_INFO "[my_get_comm] comm=%s pid=%d\n",
	       comm, current->pid);

	return 0;
}

SYSCALL_DEFINE2(my_print_user_msg, const char __user *, buf, size_t, len)
{
	char kbuf[64];

	if (!buf) {
		printk(KERN_INFO "[my_print_user_msg] error: buf is NULL\n");
		return -EINVAL;
	}

	if (len == 0 || len >= sizeof(kbuf)) {
		printk(KERN_INFO "[my_print_user_msg] error: invalid len=%zu\n", len);
		return -EINVAL;
	}

	memset(kbuf, 0, sizeof(kbuf));

	if (copy_from_user(kbuf, buf, len)) {
		printk(KERN_INFO "[my_print_user_msg] error: copy_from_user failed\n");
		return -EFAULT;
	}

	kbuf[len] = '\0';

	printk(KERN_INFO "[my_print_user_msg] comm=%s pid=%d msg=\"%s\"\n",
	       current->comm, current->pid, kbuf);

	return 0;
}
```

```c
// test_syscall.c

#define _GNU_SOURCE
#include <unistd.h>
#include <sys/syscall.h>
#include <stdio.h>
#include <errno.h>
#include <string.h>

#define SYS_my_add             549
#define SYS_my_task_info       550
#define SYS_my_get_comm        551
#define SYS_my_print_user_msg  552

static void test_my_add(void)
{
	long ret;

	errno = 0;
	ret = syscall(SYS_my_add, 10, 20);

	if (ret < 0) {
		printf("[user] my_add failed: ret=%ld errno=%d %s\n",
		       ret, errno, strerror(errno));
		return;
	}

	printf("[user] my_add ret = %ld\n", ret);
}

static void test_my_task_info(void)
{
	long ret;

	errno = 0;
	ret = syscall(SYS_my_task_info);

	if (ret < 0) {
		printf("[user] my_task_info failed: ret=%ld errno=%d %s\n",
		       ret, errno, strerror(errno));
		return;
	}

	printf("[user] my_task_info ret pid = %ld\n", ret);
}

static void test_my_get_comm_ok(void)
{
	long ret;
	char comm[16];

	memset(comm, 0, sizeof(comm));

	errno = 0;
	ret = syscall(SYS_my_get_comm, comm, sizeof(comm));

	if (ret < 0) {
		printf("[user] my_get_comm failed: ret=%ld errno=%d %s\n",
		       ret, errno, strerror(errno));
		return;
	}

	printf("[user] my_get_comm ret = %ld, comm = %s\n", ret, comm);
}

static void test_my_get_comm_small_len(void)
{
	long ret;
	char comm[4];

	memset(comm, 0, sizeof(comm));

	errno = 0;
	ret = syscall(SYS_my_get_comm, comm, sizeof(comm));

	if (ret < 0) {
		printf("[user] my_get_comm small len failed as expected: ret=%ld errno=%d %s\n",
		       ret, errno, strerror(errno));
		return;
	}

	printf("[user] unexpected success: ret=%ld comm=%s\n", ret, comm);
}

static void test_my_get_comm_bad_pointer(void)
{
	long ret;

	errno = 0;
	ret = syscall(SYS_my_get_comm, (char *)0x1, 16);

	if (ret < 0) {
		printf("[user] my_get_comm bad pointer failed as expected: ret=%ld errno=%d %s\n",
		       ret, errno, strerror(errno));
		return;
	}

	printf("[user] unexpected success: ret=%ld\n", ret);
}

static void test_my_print_user_msg_ok(void)
{
	long ret;
	const char *msg = "hello kernel";

	errno = 0;
	ret = syscall(SYS_my_print_user_msg, msg, strlen(msg));

	if (ret < 0) {
		printf("[user] my_print_user_msg failed: ret=%ld errno=%d %s\n",
		       ret, errno, strerror(errno));
		return;
	}

	printf("[user] my_print_user_msg ret = %ld\n", ret);
}

static void test_my_print_user_msg_bad_pointer(void)
{
	long ret;

	errno = 0;
	ret = syscall(SYS_my_print_user_msg, (char *)0x1, 12);

	if (ret < 0) {
		printf("[user] my_print_user_msg bad pointer failed as expected: ret=%ld errno=%d %s\n",
		       ret, errno, strerror(errno));
		return;
	}

	printf("[user] unexpected success: ret=%ld\n", ret);
}

int main(void)
{
	printf("===== syscall lab test start =====\n");

	test_my_add();
	test_my_task_info();
	test_my_get_comm_ok();
	test_my_get_comm_small_len();
	test_my_get_comm_bad_pointer();
	test_my_print_user_msg_ok();
	test_my_print_user_msg_bad_pointer();

	printf("===== syscall lab test end =====\n");

	return 0;
}
```

# 四、关键函数解析

## 1. `SYSCALL_DEFINE2(my_add, int, a, int, b)`

```c
SYSCALL_DEFINE2(my_add, int, a, int, b)
{
	int result = a + b;

	printk(KERN_INFO "[my_add] current=%s pid=%d a=%d b=%d result=%d\n",
	       current->comm, current->pid, a, b, result);

	return result;
}
```

功能：

```text
接收用户态传入的两个整数 a 和 b
在内核中计算 a + b
通过 return 把结果返回给用户态
通过 printk 打印当前调用进程的信息
```

用户态调用：

```c
syscall(549, 10, 20);
```

内核中参数对应关系：

```text
a = 10
b = 20
result = 30
```

学习点：

```text
普通值参数可以直接通过寄存器传入内核
系统调用的返回值通过 return long 返回用户态
```

------

## 2. `SYSCALL_DEFINE0(my_task_info)`

```c
SYSCALL_DEFINE0(my_task_info)
{
	printk(KERN_INFO "[my_task_info] comm=%s pid=%d tgid=%d\n",
	       current->comm, current->pid, current->tgid);

	return current->pid;
}
```

功能：

```text
打印当前调用系统调用的进程名、pid 和 tgid
返回当前进程 pid
```

关键点是 `current`。

`current` 表示：

```text
当前正在 CPU 上运行的 task_struct
```

在系统调用场景中，`current` 通常就是发起 syscall 的那个用户进程。

例如用户程序是：

```bash
./test_syscall
```

那么在内核系统调用中：

```c
current->comm
```

通常就是：

```text
test_syscall
```

这里要特别注意：

```text
系统调用不会创建新进程。
它只是当前用户进程从用户态切换到内核态执行一段内核代码。
执行完成后，再返回用户态继续运行。
```

## 3. `SYSCALL_DEFINE2(my_get_comm, char __user *, buf, size_t, len)`

```c
SYSCALL_DEFINE2(my_get_comm, char __user *, buf, size_t, len)
{
	char comm[TASK_COMM_LEN];

	if (!buf) {
		printk(KERN_INFO "[my_get_comm] error: buf is NULL\n");
		return -EINVAL;
	}

	if (len < TASK_COMM_LEN) {
		printk(KERN_INFO "[my_get_comm] error: len=%zu is too small, need %d\n",
		       len, TASK_COMM_LEN);
		return -EINVAL;
	}

	get_task_comm(comm, current);

	if (copy_to_user(buf, comm, TASK_COMM_LEN)) {
		printk(KERN_INFO "[my_get_comm] error: copy_to_user failed\n");
		return -EFAULT;
	}

	printk(KERN_INFO "[my_get_comm] comm=%s pid=%d\n",
	       comm, current->pid);

	return 0;
}
```

功能：

```text
获取当前进程名
把当前进程名从内核空间复制到用户空间 buffer
```

用户态调用：

```c
char comm[16];
syscall(551, comm, sizeof(comm));
```

执行流程：

```text
1. 用户态准备 char comm[16]
2. 用户态把 comm 的地址传给内核
3. 内核检查 buf 是否为空
4. 内核检查 len 是否足够
5. 内核通过 get_task_comm(comm, current) 获取当前进程名
6. 内核通过 copy_to_user 把进程名写回用户态 buffer
7. 用户态 printf 打印 comm
```

关键点：

```text
char __user *buf 表示 buf 是用户空间指针
内核不能直接 strcpy(buf, comm)
必须使用 copy_to_user
```

`copy_to_user` 方向是：

```text
内核空间 → 用户空间
```

错误处理：

```text
buf == NULL            返回 -EINVAL
len < TASK_COMM_LEN    返回 -EINVAL
用户地址不可写         返回 -EFAULT
```

用户态中会表现为：

```text
ret = -1
errno = 22 Invalid argument
```

或者：

```text
ret = -1
errno = 14 Bad address
```

## 4. `SYSCALL_DEFINE2(my_print_user_msg, const char __user *, buf, size_t, len)`

```c
SYSCALL_DEFINE2(my_print_user_msg, const char __user *, buf, size_t, len)
{
	char kbuf[64];

	if (!buf) {
		printk(KERN_INFO "[my_print_user_msg] error: buf is NULL\n");
		return -EINVAL;
	}

	if (len == 0 || len >= sizeof(kbuf)) {
		printk(KERN_INFO "[my_print_user_msg] error: invalid len=%zu\n", len);
		return -EINVAL;
	}

	memset(kbuf, 0, sizeof(kbuf));

	if (copy_from_user(kbuf, buf, len)) {
		printk(KERN_INFO "[my_print_user_msg] error: copy_from_user failed\n");
		return -EFAULT;
	}

	kbuf[len] = '\0';

	printk(KERN_INFO "[my_print_user_msg] comm=%s pid=%d msg=\"%s\"\n",
	       current->comm, current->pid, kbuf);

	return 0;
}
```

功能：

```text
从用户态读取一段字符串
复制到内核 buffer
然后通过 printk 打印
```

用户态调用：

```c
const char *msg = "hello kernel";
syscall(552, msg, strlen(msg));
```

执行流程：

```text
1. 用户态准备字符串 "hello kernel"
2. 用户态把字符串地址和长度传给内核
3. 内核检查 buf 是否为空
4. 内核检查 len 是否合理
5. 内核通过 copy_from_user 把用户态字符串复制到 kbuf
6. 内核手动补 '\0'
7. 内核通过 printk 打印字符串
```

`copy_from_user` 方向是：

```text
用户空间 → 内核空间
```

为什么要手动补 `'\0'`：

```c
kbuf[len] = '\0';
```

因为用户传入的是地址和长度，内核不能假设用户传来的字符串一定有合法的字符串结束符。

错误处理：

```text
buf == NULL       返回 -EINVAL
len == 0          返回 -EINVAL
len >= 64         返回 -EINVAL
用户地址不可读    返回 -EFAULT
```

```text
                     │
                     ▼
┌──────────────────────────────────────────────┐
│ __x64_sys_my_add / __x64_sys_my_get_comm ... │
│ 架构相关系统调用包装入口                     │
└──────────────────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────┐
│ SYSCALL_DEFINE                               │
│ 执行自定义系统调用逻辑                       │
└──────────────────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────┐
│ return long                                  │
│ 成功返回普通值，失败返回负错误码             │
└──────────────────────────────────────────────┘
                     │
                     ▼
          恢复现场，返回用户态
                     │
                     ▼
┌──────────────────────────────────────────────┐
│ 用户态获得返回值                             │
│ 成功：ret = 返回值                           │
│ 失败：ret = -1，errno 被设置                 │
└──────────────────────────────────────────────┘
```

更具体地，以 `my_get_comm` 为例：

```text
用户态：
char comm[16];
syscall(551, comm, sizeof(comm));

        ↓

syscall 指令进入内核

        ↓

entry_SYSCALL_64

        ↓

sys_call_table[551]

        ↓

__x64_sys_my_get_comm

        ↓

SYSCALL_DEFINE2(my_get_comm, char __user *, buf, size_t, len)

        ↓

检查 buf
检查 len
get_task_comm(comm, current)
copy_to_user(buf, comm, TASK_COMM_LEN)

        ↓

return 0

        ↓

用户态 printf("comm = %s\n", comm);
```

# 六、内核的行为分析

## 1. `printk` 打印到哪里

内核中使用：

```c
printk(KERN_INFO "message\n");
```

它不会像用户态 `printf` 一样直接打印到标准输出。

`printk` 输出到：

```text
内核日志缓冲区
```

查看方式：

```bash
dmesg
```

或者：

```bash
dmesg | tail -50
```

区别：

```text
printf   用户态输出，直接显示在终端
printk   内核态输出，进入内核日志，用 dmesg 查看
```

------

## 2. syscall 不会创建新进程

用户态执行：

```c
syscall(549, 10, 20);
```

不会创建一个新的内核进程。

真实过程是：

```text
同一个 test_syscall 进程
先在用户态执行
遇到 syscall 指令后切换到内核态
在内核态执行系统调用代码
执行结束后返回用户态
继续执行 test_syscall 的下一行代码
```

也就是说：

```text
系统调用 = 当前进程进入内核，请内核帮它完成操作
```

不是：

```text
用户进程 + 新内核进程
```

而是：

```text
同一个进程的用户态执行阶段和内核态执行阶段
```

------

## 3. `current` 在系统调用中的含义

`current` 是内核中的宏，表示：

```text
当前正在 CPU 上运行的 task_struct
```

在本实验中，用户态程序 `test_syscall` 发起系统调用后，进入内核执行时：

```c
current->comm
```

就是：

```text
test_syscall
current->pid
```

就是当前 `test_syscall` 进程的 pid。

所以在系统调用中：

```c
printk(KERN_INFO "current=%s pid=%d\n",
       current->comm, current->pid);
```

打印的是发起 syscall 的那个用户进程的信息。

这说明：

```text
内核代码是在当前进程上下文中执行的。
```

------

## 4. 为什么用户态和内核态之间要用 copy 函数

普通整数参数可以直接使用，例如：

```c
syscall(549, 10, 20);
```

内核中：

```c
SYSCALL_DEFINE2(my_add, int, a, int, b)
```

这里 `a` 和 `b` 是值参数，可以直接使用。

但是指针参数不一样，例如：

```c
char comm[16];
syscall(551, comm, sizeof(comm));
```

这里传给内核的是用户空间地址。

在内核中：

```c
char __user *buf
```

表示：

```text
buf 是用户空间指针
不能直接解引用
```

错误写法：

```c
strcpy(buf, comm);
```

正确写法：

```c
copy_to_user(buf, comm, TASK_COMM_LEN);
```

原因是用户态指针不可信：

```text
1. 用户可能传 NULL
2. 用户可能传 0x1 这种非法地址
3. 用户可能传不可写地址
4. 用户可能传长度不足的 buffer
5. 用户空间页可能没有映射
```

因此内核必须使用：

```c
copy_to_user()
copy_from_user()
```

它们可以安全地处理用户空间地址访问。

方向记忆：

```text
copy_to_user      内核 → 用户
copy_from_user    用户 → 内核
```

------

## 5. 错误码如何返回用户态

内核中返回：

```c
return -EINVAL;
```

用户态通过 `syscall()` 看到：

```text
ret = -1
errno = 22
strerror(errno) = Invalid argument
```

内核中返回：

```c
return -EFAULT;
```

用户态看到：

```text
ret = -1
errno = 14
strerror(errno) = Bad address
```

本实验中两个主要错误码：

| 内核错误码 | errno | 含义             | 场景                                                 |
| ---------- | ----- | ---------------- | ---------------------------------------------------- |
| `-EINVAL`  | 22    | Invalid argument | 参数无效，例如长度太小                               |
| `-EFAULT`  | 14    | Bad address      | 用户地址非法，`copy_to_user` / `copy_from_user` 失败 |

# 七、实验中遇到的问题与解决

## 问题：文件存在，但执行时报 `not found`

现象：

```bash
./test_syscall
/bin/sh: ./test_syscall: not found
```

同时 `ls` 可以看到：

```text
test_syscall
```

原因：

```text
test_syscall 是动态链接程序
initramfs 中没有动态链接器或 glibc 运行库
```

解决方法：

```bash
gcc -static test_syscall.c -o test_syscall
```

然后重新复制到 initramfs 并打包：

```bash
cp test_syscall initramfs/
cd initramfs
find . -print0 | cpio --null -ov --format=newc > ../initramfs.cpio
```

这个问题说明：

```text
在极简 initramfs 环境中，用户态测试程序最好静态编译。
```

# 八、本实验学到了什么

通过本实验，可以把系统调用的核心链路串起来：

```text
用户态 syscall()
        ↓
syscall 指令
        ↓
CPU 从用户态切换到内核态
        ↓
entry_SYSCALL_64
        ↓
读取 syscall number
        ↓
sys_call_table 查表
        ↓
进入对应 SYSCALL_DEFINE 实现
        ↓
执行内核逻辑
        ↓
return long
        ↓
返回用户态
```

具体收获如下：

## 1. 理解了系统调用号的作用

用户态并不是直接调用内核函数，而是通过系统调用号：

```c
syscall(549, 10, 20);
```

内核根据 `549` 在系统调用表中找到对应的处理函数。

------

## 2. 理解了 `syscall_64.tbl` 的作用

`syscall_64.tbl` 是 x86_64 架构下系统调用号和系统调用函数之间的映射表。

实验中添加：

```text
549	common	my_add		sys_my_add
```

就相当于告诉内核：

```text
549 号系统调用对应 my_add
```

------

## 3. 理解了 `SYSCALL_DEFINE` 的作用

系统调用不建议直接手写普通函数：

```c
asmlinkage long sys_my_add(...)
```

更推荐使用：

```c
SYSCALL_DEFINE2(my_add, int, a, int, b)
```

原因是 `SYSCALL_DEFINE` 会配合内核系统调用机制生成相关包装函数和元信息。

------

## 4. 理解了 `current` 的含义

在系统调用中：

```c
current->comm
current->pid
```

表示当前发起系统调用的进程信息。

这说明系统调用是在当前进程上下文中执行的，不会额外创建一个新进程。

------

## 5. 理解了用户态和内核态的数据边界

普通参数可以直接传递：

```c
int a
int b
size_t len
```

但是用户空间指针不能直接访问：

```c
char __user *buf
const char __user *buf
```

必须使用：

```c
copy_to_user()
copy_from_user()
```

这体现了用户态和内核态之间的安全边界。

------

## 6. 理解了内核错误码和用户态 errno 的关系

内核返回：

```c
return -EINVAL;
return -EFAULT;
```

用户态看到：

```text
ret = -1
errno = 22 / 14
```

也就是说，系统调用失败时，内核返回负错误码，glibc 的 `syscall()` 会把它转换成 `-1` 并设置 `errno`。

------

## 7. 理解了 `printk` 和 `printf` 的区别

```text
printf  是用户态输出
printk  是内核态日志输出
```

查看 `printk` 输出需要使用：

```bash
dmesg
```

------

# 九、实验总结

本实验通过添加 4 个自定义系统调用，完整验证了 Linux 系统调用从用户态进入内核态、执行内核逻辑、再返回用户态的基本过程。

其中：

```text
my_add
```

验证了普通参数传递和返回值。

```text
my_task_info
```

验证了系统调用中的 `current` 表示当前调用进程。

```text
my_get_comm
```

验证了内核向用户态返回数据时必须使用 `copy_to_user`。

```text
my_print_user_msg
```

验证了用户态向内核传递数据时必须使用 `copy_from_user`。

最终，本实验建立了对系统调用机制的基本认识：

```text
系统调用不是创建新进程，
而是当前用户进程通过 syscall 指令进入内核态，
由内核根据系统调用号查表，
执行对应内核函数，
再把结果返回给用户态。
```

这一实验已经覆盖了系统调用模块 80% 的核心内容。
