# 1. xv6 的进程模型

## 1.1 一个进程由两部分组成

### ① 用户态部分（User space）

- 指令（text）
- 数据（data / bss）
- 堆（heap）
- 栈（stack）

------

### ② 内核态部分（Kernel space，仅内核可见）

- 进程状态（RUNNING / RUNNABLE / SLEEPING 等）
- 寄存器保存区（context）
- 页表
- 打开文件表
- 进程号 pid

------

✅ 用户程序只能访问自己的用户空间内存
 ✅ 进程真正的“身份”由内核中的 `struct proc` 表示

------

# 2. fork：创建新进程

## 2.1 fork 的作用

```
pid = fork();
```

`fork()` 会创建一个新的进程：

- 新进程称为子进程
- 子进程几乎复制父进程的全部内容

复制的内容包括：

- 用户内存
- 寄存器
- 打开的文件

------

但复制完成之后：

✅ 父子进程是两个完全独立的执行体

------

## 2.2 fork 的返回值规则（关键）

| 执行环境 | fork 返回值     |
| -------- | --------------- |
| 父进程   | 子进程 pid (>0) |
| 子进程   | 0               |
| 出错     | -1              |

------

常见写法：

```
if (pid > 0) {
  // 父进程
} else if (pid == 0) {
  // 子进程
}
```

------

✅ `fork()` 只调用一次，但会在两个进程中分别返回

------

## 2.3 为什么父子进程互不影响？

在 `fork()` 之后：

- 拥有不同的地址空间
- 拥有不同的寄存器状态

因此：

👉 一个进程中变量的修改，不会影响另一个进程

可以理解为：

👉 “程序被复制了一份，各自独立运行”

------

# 3. exit 和 wait：进程结束与同步

## 3.1 exit

```
exit();
```

作用：

- 当前进程立即结束执行
- 释放占用的资源（内存等）
- 通知内核该进程已经退出

------

## 3.2 wait

```
pid = wait();
```

行为：

- 如果已有子进程退出 → 立即返回其 pid
- 如果没有 → 父进程进入睡眠，直到子进程退出

------

✅ `wait()` 的作用：

- 实现父子进程同步
- 避免产生僵尸进程

------

# 4. exec：替换进程中的程序（重点）

## 4.1 exec 的本质（一定要记住）

❗ `exec` 不会创建新进程
 ❗ 它只是替换当前进程的用户空间内容

------

`exec` 做的事情：

- 清空当前进程的用户内存（代码、数据、栈）
- 从可执行文件加载新的程序（ELF）
- 构造新的用户栈（argc / argv）
- 从 ELF 指定的入口地址开始执行

------

## 4.2 exec 成功后发生什么？

```
exec("/bin/echo", argv);
printf("exec error\n");
```

- 如果 `exec` 成功
   👉 原程序被完全替换
   👉 后面的代码不会再执行
- 如果 `exec` 失败
   👉 才会执行 `printf("exec error\n")`

------

✅ 成功后：

👉 原程序“消失”，新程序从头开始运行

------

## 4.3 exec 的参数

```
exec(char *path, char **argv);
```

示例：

```
argv[0] = "echo";
argv[1] = "hello";
argv[2] = 0;
```

说明：

- `argv[0]` 按约定表示程序名
- xv6 / Unix 程序通常不会严格依赖它

------

# 5. shell 是如何执行命令的？

## 5.1 shell 的基本流程

用户输入：

```
echo hello
```

------

## 5.2 shell 内部实际发生的事情

```
shell
 |-- fork()
 |    |-- 父进程：wait()
 |    |
 |    |-- 子进程：exec("echo")
 |               |
 |             echo 运行
 |               |
 |             exit()
 |-- 父进程 wait 返回
 |-- shell 继续运行
```

------

### 分工说明

父进程（shell）：

- 调用 `wait()`
- 等待子进程结束

子进程：

- 调用 `exec()`
- 变成用户要执行的程序（如 echo）

------

## 5.3 为什么 shell 自身不会被替换？

因为：

- shell 在 `fork()` 后
- 只让子进程执行 `exec()`
- 自己只负责等待和循环

------

✅ shell 始终存在
 ✅ 用户程序只是短暂的子进程

------

# 6. 为什么 fork 和 exec 不合并？（核心设计思想）

## 6.1 如果 fork + exec 合并，会有什么问题？

在 shell 中，`fork` 和 `exec` 之间需要做额外操作，例如：

- I/O 重定向（>、<）
- 管道（|）
- 后台执行（&）

------

示例：

```
echo hello > out.txt
```

实际执行逻辑：

```
fork();
if (child) {
  close(1);
  open("out.txt");
  exec("echo");
}
wait();
```

------

如果把 `fork` 和 `exec` 合成一个操作：

⛔ shell 将无法在中间插入这些控制逻辑

------

## 6.2 结论

👉 fork 与 exec 分离
 是 Unix 设计中非常经典且关键的一点

------

# 7. 一句话总结（务必记住）

```
fork：复制一个进程
exec：在当前进程中替换程序
shell：fork 子进程，让子进程 exec
```
