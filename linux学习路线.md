# 一、先说总路线

我建议你按这 4 个阶段练：

## 第 1 阶段：只做“换内核”

目标是：

- 会下载内核源码
- 会配置和编译
- 会安装/启动新内核
- 会切回旧内核

这阶段先不追求源码理解很深，先把流程跑通。

------

## 第 2 阶段：改“最容易验证”的源码

目标是：

- 改一点源码
- 编译
- 启动
- 看到结果确实变了

这一步是建立信心的关键。

------

## 第 3 阶段：改“内核行为”

目标是：

- 不只是改打印字符串
- 而是改变一点系统行为
- 比如新增一个 `/proc` 项，或者改一个简单策略

------

## 第 4 阶段：做小型内核项目

目标是：

- 把几个点串起来
- 形成一个完整练习项目
- 最后能写到简历或者学习笔记里

------

# 二、最经典的入手路线

下面这条路线，是我真心建议你按顺序做的。

------

## 路线 0：准备环境

先在 **虚拟机/QEMU** 里做，不要先拿主力系统开刀。

建议环境：

- Ubuntu 虚拟机
- 保留原内核可回退
- 学会查看当前内核版本：

```
uname -r
```

------

## 路线 1：编译并启动一个“原版新内核”

这是第一关。

### 目标

你什么都不改，先把一个新内核编出来并启动成功。

### 你会练到

- 获取源码
- `make menuconfig`
- `make`
- 安装模块
- 安装内核
- 更新 bootloader
- 重启选择新内核
- 验证版本变化

### 验证方式

重启后执行：

```
uname -r
```

看到变成你编译的新版本。

### 这一关的意义

这是“换内核”的基本功。
 很多人卡死在这一步，后面根本谈不上改源码。

------

## 路线 2：改内核版本后缀或启动标识

这是最经典的新手第一改。

### 做法

改内核配置或者源码，让启动后内核版本里带上你自己的标记。

比如让 `uname -r` 变成：

- `6.x.x-mytest`

或者在启动日志里加一句：

- `Hello from modified kernel`

### 为什么经典

因为它：

- 改动非常小
- 很容易验证
- 不容易把系统搞挂
- 能建立“我真的改动了内核”的直觉

### 你会练到

- 改源码/配置
- 重新编译
- 验证改动是否生效
- 看 `dmesg`

------

## 路线 3：修改启动日志（printk）

这是第二个非常经典的练习。

### 做法

在内核启动流程某个合适位置加一句 `printk()`。

比如在启动阶段打印：

- 当前 CPU 信息
- 自定义字符串
- 某个内核初始化阶段的提示

### 你会练到

- 找入口点
- 看内核启动主线
- 用 `dmesg` 验证
- 理解“代码改动 → 启动日志变化”

### 为什么适合入门

因为反馈非常快，改完一重启就能看到。

------

## 路线 4：增加一个 `/proc` 节点

这是我特别推荐你的第一个“像样内核练习”。

### 做法

做一个简单的 `/proc/mytest`，读取时返回一段你定义的内容，比如：

- 内核版本
- jiffies
- 一个固定字符串
- 当前时间片计数之类

### 你会练到

- procfs 接口
- 内核里如何导出信息给用户态
- 文件读路径
- 用户态 `cat /proc/mytest` 验证

### 为什么非常推荐

因为它开始把：

- 内核空间
- 用户空间
- 文件接口
   串起来了。

而且风险很低。

------

## 路线 5：写一个内核模块，而不是改大内核

这个也很经典，而且很实用。

### 做法

先写一个简单模块：

- `insmod`
- `rmmod`
- `dmesg` 看日志

再升级成：

- 模块参数
- `/proc` 或 `/sys` 交互
- 定时器/workqueue 小实验

### 为什么这一步很重要

因为现实里很多内核开发入门，不是直接魔改内核主线，而是先从模块练起。

### 你会练到

- 内核构建系统
- 模块生命周期
- 内核日志
- 用户态与内核态交互

------

## 路线 6：做一个简单字符设备驱动

这是比 `/proc` 更进一步的经典路线。

### 做法

实现一个简单 char device，支持：

- open
- read
- write
- 也许再加一个 ioctl

比如用户执行：

```
echo hello > /dev/mydev
cat /dev/mydev
```

### 你会练到

- file_operations
- 设备号
- 用户态和内核态数据拷贝
- 驱动基本骨架

### 这是个分水岭

做到这一步，你已经不只是“改内核玩”，而是在开始理解 Linux 内核接口了。

------

## 路线 7：改一个简单系统调用或加一个新系统调用

这个很经典，但我建议放后面。

### 做法

比如加一个很简单的 syscall：

- 返回一个固定值
- 返回当前 jiffies
- 返回某个统计信息

### 为什么不建议太早做

因为它会牵涉：

- syscall table
- 架构相关入口
- 用户态测试程序
- ABI/头文件

对新手来说，比 `/proc`、模块、char device 更绕。

但它确实是经典练习。

------

## 路线 8：改调度器/内存管理？先别急

很多人入门就想改：

- scheduler
- page allocator
- virtual memory
- CFS 行为

这个方向当然值钱，但**不适合最开始**。
 因为这些地方：

- 代码复杂
- 改坏了很难定位
- 很容易导致系统不稳定
- 验证成本高

你应该等前面的路线都走过，再选一个深挖。

------

# 三、我给你一个“经典入手顺序”

如果只选最稳的一条，我建议：

## 第一阶段

1. 编译并启动新内核
2. 改版本后缀/启动标识
3. 改 `printk` 启动日志

## 第二阶段

1. 做一个 `/proc/mytest`
2. 写 hello module
3. 模块参数 + `/proc`/`sysfs`

## 第三阶段

1. 简单 char device
2. 增加一个简单 syscall

## 第四阶段

1. 选一个子系统深入：

- 调度
- 内存
- VFS
- 中断
- 驱动模型

这条路线很稳，也非常经典。

# 四、每一步你到底在练什么

你别把这些题目看成“杂活”，它们其实各练一项核心能力。

### 编译换内核

练：

- 环境
- 构建
- 启动
- 回滚

### 改 printk / 版本号

练：

- 最小改动验证
- 日志观察
- 改动闭环

### `/proc` 节点

练：

- 内核导出信息
- 用户态接口

### 模块

练：

- 内核开发的低风险入口
- 生命周期和调试

### 字符设备

练：

- Linux 驱动基本模型
- file_operations

### syscall

练：

- 用户态到内核态入口链路

# 五、你说“得有个现成题目或者目的”，我给你几个很经典的练习题

下面这些都很适合做成“今晚/本周的小目标”。

------

## 题目 1：给内核加自己的版本标记

### 目标

让 `uname -r` 带上你的后缀。

### 验证

```
uname -r
```

------

## 题目 2：启动时打印一句你自己的日志

### 目标

在 `dmesg` 里能搜到自定义字符串。

### 验证

```
dmesg | grep your_string
```

------

## 题目 3：新增一个 `/proc/hello_kernel`

### 目标

`cat /proc/hello_kernel` 输出你定义的信息。

### 验证

```
cat /proc/hello_kernel
```

------

## 题目 4：写一个 hello 模块

### 目标

加载模块打印 “module loaded”，卸载打印 “module unloaded”。

### 验证

```
sudo insmod xxx.ko
dmesg
sudo rmmod xxx
dmesg
```

------

## 题目 5：写一个带参数的模块

### 目标

加载时传参数：

```
sudo insmod xxx.ko myvalue=123
```

日志里打印出这个值。

------

## 题目 6：实现一个简单字符设备

### 目标

通过 `/dev/mydev` 读写内核中的一段缓冲区。

------

## 题目 7：做一个“系统状态观测器”

### 目标

通过 `/proc` 或模块输出：

- 当前 jiffies
- 内存页统计
- 进程数
- uptime

这类项目非常适合入门。

------

# 六、如果你想“给 Linux 换个内核”，最现实的练法是什么

我建议你这么练，而不是直接动宿主机：

## 最好在虚拟机里：

- 保留原内核
- 编新内核
- 从 grub 选新内核启动
- 出问题还能退回旧内核

这样容错高很多。

### 最低要求

你要保证：

- 至少保留一个可用旧内核
- 不要覆盖掉唯一可启动项
- 每次改动都能回滚

------

# 七、你现在最适合哪条路线

结合你现在情况：已经工作、目标偏底层、但刚开始系统学内核。

我建议你**不要一开始改调度器和内存管理**，也不要先追 syscall。
 你现在最适合的是：

## 近 4 步主线

1. 在虚拟机里编译并切换到新内核
2. 改内核版本后缀 / 启动日志
3. 新增一个 `/proc` 节点
4. 写一个简单内核模块

这 4 步走完，你会一下子明白很多事：

- 内核是怎么构建的
- 改动怎么验证
- 用户态怎么观察内核
- 模块和内核本体区别是什么

------

# 九、我给你的直话

你现在最需要的不是“更高级的题目”，而是**能闭环的小题目**。

所谓闭环，就是：

- 改了什么
- 为什么改
- 编译了
- 启动了
- 验证了
- 确认生效

很多人学内核学不下去，不是因为题目不够高级，而是因为一开始就选太大、太难、太难验证的目标。
