# 内联函数小笔记

## 1. 什么是内联函数

内联函数就是：**编译器在编译时，尽量把函数体直接展开到调用位置，而不是生成一次普通函数调用。**

普通函数调用：

```c
int add(int a, int b)
{
    return a + b;
}

int x = add(10, 20);
```

执行逻辑大概是：

```text
跳到 add()
执行 a + b
返回调用位置
```

如果被内联，编译器可能直接优化成：

```c
int x = 10 + 20;
```

也就是说，函数体被“复制”到了调用处。

------

## 2. 为什么要用内联函数

普通函数调用有一定开销，例如：

```text
参数传递
跳转到函数地址
保存/恢复现场
函数返回
```

如果函数本身很短，但调用非常频繁，函数调用开销可能比函数逻辑本身还明显。

例如：

```c
static inline int max(int a, int b)
{
    return a > b ? a : b;
}
```

这种函数逻辑很短，适合内联。

------

## 3. 基本定义方式

常见写法：

```c
static inline int add(int a, int b)
{
    return a + b;
}
```

含义：

```text
static：限制函数作用域，避免多个文件包含时重复定义
inline：建议编译器把函数体展开到调用处
```

注意：`inline` 只是建议，不是强制。编译器可能内联，也可能不内联。

------

## 4. Linux 内核里的写法

在 Linux 内核中，经常能看到：

```c
static __always_inline unsigned long __must_check
copy_to_user(void __user *to, const void *from, unsigned long n)
{
	if (check_copy_size(from, n, true))
		n = _copy_to_user(to, from, n);
	return n;
}
```

这里的几个关键词：

```text
static
```

表示这个函数只在当前编译单元内可见，适合写在头文件里。

```text
__always_inline
```

表示尽量强制编译器内联，比普通 `inline` 更强。

```text
__must_check
```

表示调用者必须检查返回值。比如 `copy_to_user()` 可能失败，不能忽略返回值。

------

## 5. 内联函数怎么用

定义：

```c
static inline int square(int x)
{
    return x * x;
}
```

使用：

```c
int a = 5;
int b = square(a);
```

编译器可能把它优化成：

```c
int b = a * a;
```

从使用方式看，内联函数和普通函数基本一样：

```text
定义函数
传参数
拿返回值
```

区别主要发生在编译阶段。

------

## 6. 内联函数适合什么情况

适合使用内联函数的情况：

```text
1. 函数很短
2. 调用非常频繁
3. 函数逻辑简单
4. 不希望频繁产生函数调用开销
5. 一些头文件中的小工具函数
6. 内核底层包装函数
```

例如：

```c
static inline int is_power_on(int status)
{
    return status & 0x1;
}
```

这种位判断函数很短，适合内联。

------

## 7. 不适合使用内联函数的情况

不适合内联的情况：

```text
1. 函数体很大
2. 函数逻辑复杂
3. 调用次数不多
4. 包含大量循环、分支、复杂处理
5. 递归函数
```

原因是：内联会把函数体复制到每个调用点。如果函数很大、调用点很多，代码体积会明显变大。

例如不太适合：

```c
static inline void big_func(void)
{
    // 很多复杂逻辑
    // 很多循环
    // 很多分支
}
```

这种函数强行内联，可能得不偿失。

------

## 8. 内联函数的优点

```text
1. 减少函数调用开销
2. 对短小频繁调用的函数更高效
3. 编译器可以做进一步优化
4. 比宏更安全，有类型检查
```

比如相比宏：

```c
#define SQUARE(x) ((x) * (x))
```

内联函数更安全：

```c
static inline int square(int x)
{
    return x * x;
}
```

因为函数有明确的参数类型和返回类型。

------

## 9. 内联函数的缺点

```text
1. 可能增加代码体积
2. 滥用会影响指令缓存
3. 大函数内联可能降低性能
4. 编译时间可能增加
5. inline 不保证一定内联
```

所以不要看到小函数就无脑加 `inline`。一般只给**短小、频繁调用、逻辑简单**的函数用。

------

## 10. 内联函数和宏的区别

| 对比项     | 宏                    | 内联函数         |
| ---------- | --------------------- | ---------------- |
| 展开阶段   | 预处理阶段            | 编译阶段         |
| 类型检查   | 没有                  | 有               |
| 调试友好性 | 较差                  | 较好             |
| 副作用风险 | 高                    | 低               |
| 推荐程度   | 简单常量/条件编译适合 | 小函数逻辑更适合 |

宏的风险例子：

```c
#define SQUARE(x) ((x) * (x))

int a = 3;
int b = SQUARE(a++);
```

展开后类似：

```c
int b = ((a++) * (a++));
```

`a++` 会执行两次，容易出 bug。

内联函数不会有这个问题：

```c
static inline int square(int x)
{
    return x * x;
}
```

## 11. 结合 `copy_to_user` 理解

你看到的代码：

```c
static __always_inline unsigned long __must_check
copy_to_user(void __user *to, const void *from, unsigned long n)
{
	if (check_copy_size(from, n, true))
		n = _copy_to_user(to, from, n);
	return n;
}
```

可以这样理解：

```text
copy_to_user 是一个很短的包装函数。
它先做安全检查 check_copy_size。
如果检查通过，再调用真正复制的 _copy_to_user。
因为这层函数很短、很常用，所以内核希望它被内联。
```

调用时：

```c
if (copy_to_user(buf, comm, TASK_COMM_LEN))
	return -EFAULT;
```

编译器可能直接展开成类似：

```c
if (check_copy_size(comm, TASK_COMM_LEN, true)) {
	ret = _copy_to_user(buf, comm, TASK_COMM_LEN);
} else {
	ret = TASK_COMM_LEN;
}

if (ret)
	return -EFAULT;
```

这样可以减少一次 `copy_to_user()` 包装函数的调用开销。

------

## 12. 最简单总结

```text
内联函数 = 编译器尽量把函数体展开到调用处的小函数。
```

适合：

```text
短小、频繁、简单、底层工具函数
```

不适合：

```text
大函数、复杂函数、调用不频繁的函数
```

优点：

```text
减少函数调用开销，比宏更安全
```

缺点：

```text
可能增加代码体积，不能滥用
```

在 Linux 内核中，常见形式是：

```c
static inline ...
```

或者：

```c
static __always_inline ...
```

你可以这样记：

```text
普通 inline：建议内联
__always_inline：尽量强制内联
static inline：常用于头文件里的小工具函数
```
