# 内核模块加载与卸载实验笔记

## 1. 实验目标

本实验编写一个最小 Linux 内核模块 `hello_module.ko`，通过：

```
insmod hello_module.ko
rmmod hello_module
```

验证模块可以在系统运行时动态加载和卸载。

------

## 2. 实验流程

### Step 1：创建模块目录

```
mkdir -p ~/allcode/kernel-lab/modules/hello_module
cd ~/allcode/kernel-lab/modules/hello_module
```

------

### Step 2：编写模块源码

文件：

```
hello_module.c
```

代码：

```
#include <linux/init.h>
#include <linux/module.h>
#include <linux/kernel.h>

static int __init hello_module_init(void)
{
    pr_info("hello_module: loaded\n");
    return 0;
}

static void __exit hello_module_exit(void)
{
    pr_info("hello_module: unloaded\n");
}

module_init(hello_module_init);
module_exit(hello_module_exit);

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Eric Zhou");
MODULE_DESCRIPTION("A simple hello kernel module");
```

------

### Step 3：编写 Makefile

文件：

```
Makefile
```

内容：

```
obj-m += hello_module.o

KDIR := /home/yunhe/allcode/linux-6.6.87-build

all:
	make -C $(KDIR) M=$(PWD) modules

clean:
	make -C $(KDIR) M=$(PWD) clean
```

说明：

```
obj-m += hello_module.o
```

表示把 `hello_module.c` 编译成外部模块，最终生成：

```
hello_module.ko
KDIR := /home/yunhe/allcode/linux-6.6.87-build
```

表示使用已经编译好的内核构建目录。

```
make -C $(KDIR) M=$(PWD) modules
```

表示进入内核构建环境，用当前目录源码编译外部模块。

------

### Step 4：编译模块

```
make
```

成功后生成：

```
hello_module.ko
```

------

### Step 5：把模块放入 initramfs

因为 QEMU 中的最小系统无法直接访问宿主机文件，所以需要把 `.ko` 放入 initramfs：

```
cp hello_module.ko ~/allcode/kernel-lab/initramfs/root/
```

如果没有 `root` 目录：

```
mkdir -p ~/allcode/kernel-lab/initramfs/root
cp hello_module.ko ~/allcode/kernel-lab/initramfs/root/
```

然后重新打包：

```
cd ~/allcode/kernel-lab
./scripts/pack.sh
```

------

### Step 6：启动 QEMU

```
qemu-system-x86_64 \
  -kernel ~/allcode/linux-6.6.87-build/arch/x86/boot/bzImage \
  -initrd ~/allcode/kernel-lab/initramfs.cpio \
  -append "console=ttyS0 rdinit=/init" \
  -nographic
```

------

### Step 7：加载模块

进入 QEMU 后：

```
insmod /root/hello_module.ko
```

查看日志：

```
dmesg | tail
```

看到：

```
hello_module: loading out-of-tree module taints kernel.
hello_module: loaded
```

说明模块加载成功，`hello_module_init()` 被执行。

------

### Step 8：卸载模块

```
rmmod hello_module
```

查看日志：

```
dmesg | tail
```

看到：

```
hello_module: unloaded
```

说明模块卸载成功，`hello_module_exit()` 被执行。

------

## 3. 代码解析

### 3.1 头文件

```
#include <linux/init.h>
#include <linux/module.h>
#include <linux/kernel.h>
```

`linux/init.h` 提供：

```
__init
__exit
module_init()
module_exit()
```

`linux/module.h` 提供模块相关机制和模块元信息宏，例如：

```
MODULE_LICENSE()
MODULE_AUTHOR()
MODULE_DESCRIPTION()
```

`linux/kernel.h` 提供内核常用函数，例如：

```
pr_info()
```

------

### 3.2 模块加载函数

```
static int __init hello_module_init(void)
{
    pr_info("hello_module: loaded\n");
    return 0;
}
```

函数类型：

```
static int __init
```

含义：

- `static`：函数只在当前 `.c` 文件内可见
- `int`：返回加载结果
- `__init`：表示这是初始化函数，只在模块加载阶段使用

输入参数：

```
void
```

表示没有输入参数。

返回值：

```
return 0;
```

表示模块加载成功。

如果返回负数，表示模块加载失败。

作用：

当执行：

```
insmod hello_module.ko
```

内核会调用这个函数。

------

### 3.3 模块卸载函数

```
static void __exit hello_module_exit(void)
{
    pr_info("hello_module: unloaded\n");
}
```

函数类型：

```
static void __exit
```

含义：

- `static`：函数只在当前文件内可见
- `void`：没有返回值
- `__exit`：表示这是退出函数，只在模块卸载阶段使用

输入参数：

```
void
```

表示没有输入参数。

返回值：

无。

作用：

当执行：

```
rmmod hello_module
```

内核会调用这个函数。

------

### 3.4 `module_init()`

```
module_init(hello_module_init);
```

作用：

告诉内核：

```
加载这个模块时，执行 hello_module_init()
```

也就是说：

```
insmod hello_module.ko
```

会触发：

```
hello_module_init()
```

------

### 3.5 `module_exit()`

```
module_exit(hello_module_exit);
```

作用：

告诉内核：

```
卸载这个模块时，执行 hello_module_exit()
```

也就是说：

```
rmmod hello_module
```

会触发：

```
hello_module_exit()
```

------

### 3.6 模块元信息

```
MODULE_LICENSE("GPL");
MODULE_AUTHOR("Eric Zhou");
MODULE_DESCRIPTION("A simple hello kernel module");
```

`MODULE_LICENSE("GPL")` 声明模块许可证。

如果不写，内核可能提示模块 license 不明，并 taint kernel。

`MODULE_AUTHOR()` 声明作者信息。

`MODULE_DESCRIPTION()` 声明模块描述。

这些信息可以通过：

```
modinfo hello_module.ko
```

查看。

## 4. 内核在加载和卸载模块时做了什么

### 4.1 `insmod` 加载模块流程

用户执行：

```
insmod /root/hello_module.ko
```

大致流程：

```
用户态 insmod
  ↓
读取 hello_module.ko 文件
  ↓
通过系统调用请求内核加载模块
  ↓
内核解析 .ko 文件
  ↓
检查模块格式、版本、符号依赖
  ↓
为模块分配内核内存
  ↓
把模块代码和数据加载进内核空间
  ↓
完成符号重定位
  ↓
调用 module_init() 注册的 hello_module_init()
  ↓
打印 hello_module: loaded
  ↓
模块进入 loaded 状态
```

也就是说，`insmod` 只是用户态命令，真正把模块加载进内核的是内核模块加载机制。

------

### 4.2 `rmmod` 卸载模块流程

用户执行：

```
rmmod hello_module
```

大致流程：

```
用户态 rmmod
  ↓
请求内核卸载 hello_module
  ↓
内核检查模块是否正在被使用
  ↓
如果引用计数为 0，允许卸载
  ↓
调用 module_exit() 注册的 hello_module_exit()
  ↓
打印 hello_module: unloaded
  ↓
释放模块占用的内核资源
  ↓
模块从内核中移除
```

------

## 5. 为什么会看到 taints kernel

日志中出现：

```
hello_module: loading out-of-tree module taints kernel.
```

这不是错误。

意思是：

```
当前加载的是源码树外编译的外部模块
```

内核会把自己标记为 tainted，表示当前内核已经不是完全“原始、纯净”的状态。

这个提示对实验没有影响，可以忽略。

## 6. 这个实验的意义

这个实验完成了从“静态改内核”到“动态扩展内核”的过渡。

之前的实验流程是：

```
改内核源码
  ↓
重新编译整个内核
  ↓
重启 QEMU
  ↓
验证结果
```

这条链路很重。

模块实验的流程是：

```
写模块源码
  ↓
编译成 .ko
  ↓
insmod 加载
  ↓
dmesg 验证
  ↓
rmmod 卸载
```

不需要重新编译整个内核，也不需要重启系统。

## 7. 本实验学到的核心概念

### 7.1 内核模块是什么

内核模块是可以在运行时加载进内核的功能代码。

它不是用户态程序，不能直接运行：

```
./hello_module.ko
```

必须通过：

```
insmod hello_module.ko
```

加载进内核。

------

### 7.2 模块和内核的关系

内核是主体，模块是可插拔功能。

```
Linux 内核 = 主程序
内核模块 = 插件
```

加载后，模块运行在内核态，可以调用内核 API。

------

### 7.3 模块和 bzImage 的关系

`bzImage` 是内核本体镜像。

模块 `.ko` 不在 `bzImage` 里，而是单独存在。

```
bzImage = 内核本体
.ko     = 可动态加载的内核功能文件
```

------

### 7.4 模块和源码文件的关系

```
hello_module.c
  ↓
hello_module.o
  ↓
hello_module.ko
  ↓
insmod 加载进内核
```

------

## 8. 一句话总结

本实验编写了一个最小 Linux 内核模块，通过 `insmod` 动态加载模块、通过 `rmmod` 动态卸载模块，并用 `dmesg` 验证模块加载函数和卸载函数的执行，掌握了内核模块的基本生命周期和动态扩展内核的开发方式。
